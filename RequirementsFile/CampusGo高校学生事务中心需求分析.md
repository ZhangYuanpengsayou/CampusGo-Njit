# CampusGo 高校学生事务中心需求分析

## 1. 项目概述

CampusGo 是一个面向高校学生事务办理的 Java Web 系统，主要用于公告浏览、用户信息管理、学生请销假、公寓报修等事务处理。系统面向游客、学生、教师、管理员四类使用者，通过角色权限控制实现不同用户的业务操作。

项目目标是为高校提供一个集中化、线上化、流程化的学生事务服务平台，减少线下审批和沟通成本，提高事务办理效率与信息透明度。

## 2. 用户角色分析

### 2.1 游客

游客指未登录系统的访问者。

主要权限：

- 浏览网站主页。
- 查看系统公告列表和公告详情。
- 进入登录、注册页面。

限制：

- 不能发布、修改、删除公告。
- 不能提交请假申请或报修申请。
- 不能查看系统内部业务数据。

### 2.2 学生

学生是系统中事务申请的主要发起者。

主要权限：

- 注册学生账号并登录系统。
- 查看、编辑、修改个人信息。
- 查询公告。
- 提交请假申请。
- 在教师审核前撤销请假申请。
- 请假结束返校后进行销假。
- 提交公寓报修申请并上传照片。
- 在管理员审核前撤销报修申请。
- 维修完成后对报修服务进行评分评价。

### 2.3 教师

教师主要负责本学院学生请假申请的查询与审核。

主要权限：

- 注册教师账号并登录系统。
- 查看、编辑、修改个人信息。
- 查询公告。
- 查询本学院学生的请假申请。
- 审核本学院学生请假申请，将申请状态修改为“通过”或“不通过”。

限制：

- 不能添加、修改、删除公告。
- 不能审核公寓报修申请。
- 原则上只能处理所属学院学生的请假数据。

### 2.4 管理员

管理员负责系统公告维护和公寓报修审核。

主要权限：

- 注册或创建管理员账号并登录系统。
- 添加公告。
- 修改公告。
- 删除公告。
- 查询所有公告。
- 查询学生公寓报修申请。
- 审核报修申请。
- 审核报修时填写维修工手机号。

限制：

- 不负责学生请假审批。
- 不应随意修改学生、教师业务申请的历史记录。

## 3. 功能需求分析

## 3.1 首页与公告模块

### 3.1.1 首页展示

系统首页应展示公告信息，方便游客、学生、教师、管理员快速获取学校事务通知。

功能要求：

- 首页显示公告列表。
- 公告列表可按发布时间倒序展示。
- 支持点击公告查看公告详情。
- 游客无需登录即可浏览公告。

### 3.1.2 公告查询

学生、教师、管理员登录后均可查询公告。

建议功能：

- 按公告标题关键字查询。
- 按发布时间范围查询。
- 查看公告标题、内容、发布人、发布时间、更新时间等信息。

### 3.1.3 公告管理

公告管理仅管理员可操作。

功能要求：

- 添加公告：填写公告标题、公告内容，发布人和发布时间由系统自动记录。
- 修改公告：管理员可修改公告标题、内容，并记录更新时间。
- 删除公告：管理员可删除不再需要展示的公告。
- 查询公告：管理员可查看公告列表和详情。

权限要求：

- 游客、学生、教师只能查看公告，不能新增、修改、删除公告。

## 3.2 用户注册、登录与个人信息模块

### 3.2.1 用户注册

系统注册应区分管理员、教师、学生三种账号类型。

功能要求：

- 注册时选择用户角色：管理员、教师、学生。
- 输入账号、密码、姓名、手机号等基础信息。
- 学生建议填写学号、学院、专业、班级、公寓信息。
- 教师建议填写工号、学院、职称或办公室信息。
- 管理员建议填写管理员编号或管理部门信息。

业务规则：

- 用户账号应唯一。
- 学号、工号等身份编号应唯一。
- 密码应进行加密存储。
- 注册成功后可登录系统。

### 3.2.2 用户登录

功能要求：

- 用户使用账号和密码登录。
- 登录成功后根据角色进入对应功能页面。
- 登录失败时提示账号或密码错误。
- 系统应保存当前登录用户信息，用于自动获取申请人等业务字段。

### 3.2.3 个人信息管理

学生、教师登录后可以编辑和修改个人信息。

功能要求：

- 查看个人资料。
- 修改姓名、手机号、邮箱、学院等允许修改的信息。
- 修改密码。

限制建议：

- 学号、工号、账号、角色等关键身份字段不建议由普通用户随意修改。
- 管理员账号信息修改可单独设置管理入口。

## 3.3 学生请销假系统

### 3.3.1 学生提交请假申请

学生登录后可填写请假申请。

填写字段：

- 申请理由。
- 开始时间。
- 结束时间。

系统自动获取字段：

- 申请人。
- 申请人所属学院。
- 申请时间。
- 申请状态，初始为“待审核”。

业务规则：

- 开始时间不能晚于结束时间。
- 请假申请提交后进入待审核状态。
- 同一时间段内不建议重复提交请假申请。

### 3.3.2 学生撤销请假申请

功能要求：

- 教师审核前，学生可以撤销请假申请。
- 已通过、已不通过、已销假的申请不能撤销。

状态变化：

- “待审核”变更为“已撤销”。

### 3.3.3 教师查询请假申请

教师登录后可查询本学院学生提交的请假申请。

查询条件建议：

- 学生姓名或学号。
- 申请状态。
- 申请时间范围。
- 请假开始时间和结束时间。

数据范围：

- 教师只能查询本学院学生的请假申请。

### 3.3.4 教师审核请假申请

教师对本学院学生的请假申请进行审核。

功能要求：

- 查看请假申请详情。
- 审核状态可修改为“通过”或“不通过”。
- 可填写审核意见。
- 系统自动记录审核人和审核时间。

业务规则：

- 只有“待审核”状态的请假申请可以审核。
- 审核后学生不能撤销。

### 3.3.5 学生销假

学生请假返校后应及时销假。

功能要求：

- 学生可对已通过的请假申请发起销假。
- 系统记录销假时间。
- 请假申请状态变更为“已销假”。

业务规则：

- 只有审核通过的申请可以销假。
- 未审核、不通过、已撤销的申请不能销假。

### 3.3.6 请假状态定义

请假申请建议包含以下状态：

- 待审核：学生提交申请后的初始状态。
- 通过：教师审核同意。
- 不通过：教师审核拒绝。
- 已撤销：教师审核前学生主动撤销。
- 已销假：学生返校后完成销假。

## 3.4 公寓报修系统

### 3.4.1 学生提交报修申请

学生登录后可提交公寓报修申请。

填写字段：

- 申请事由。
- 报修照片。

系统自动获取字段：

- 申请人。
- 申请人联系方式。
- 公寓信息。
- 申请时间。
- 申请状态，初始为“待审核”。

业务规则：

- 报修事由不能为空。
- 上传照片应限制文件类型和大小。
- 支持常见图片格式，如 jpg、jpeg、png。

### 3.4.2 学生撤销报修申请

功能要求：

- 管理员审核前，学生可以撤销报修申请。
- 已审核、维修中、已完成、已评价的申请不能撤销。

状态变化：

- “待审核”变更为“已撤销”。

### 3.4.3 管理员查询报修申请

管理员登录后可查询学生报修申请。

查询条件建议：

- 学生姓名或学号。
- 申请状态。
- 公寓楼栋或宿舍号。
- 申请时间范围。

### 3.4.4 管理员审核报修申请

管理员负责审核学生报修申请。

功能要求：

- 查看报修详情和上传照片。
- 修改报修申请状态。
- 填写维修工手机号。
- 可填写审核意见。
- 系统自动记录审核人和审核时间。

业务规则：

- 只有“待审核”状态的报修申请可以审核。
- 审核通过时必须填写维修工手机号。
- 维修工手机号应符合手机号码格式要求。

### 3.4.5 学生评分评价

维修完成后，学生可对维修服务进行评价。

功能要求：

- 学生对已完成维修的报修申请进行评分。
- 学生可填写评价内容。
- 系统记录评价时间。

业务规则：

- 只有已完成维修的报修申请可以评价。
- 同一报修申请只能评价一次。

### 3.4.6 报修状态定义

报修申请建议包含以下状态：

- 待审核：学生提交申请后的初始状态。
- 通过：管理员审核同意并分配维修工。
- 不通过：管理员审核拒绝。
- 已撤销：管理员审核前学生主动撤销。
- 维修中：维修工已被安排并开始处理。
- 已完成：维修结束。
- 已评价：学生完成评分评价。

说明：原始需求中明确要求管理员审核和填写维修工手机，维修完成后的状态流转可根据项目实现范围简化。如果课程项目不涉及维修工账号，可由管理员手动将状态更新为“已完成”。

## 4. 业务流程分析

### 4.1 公告业务流程

1. 管理员登录系统。
2. 管理员进入公告管理页面。
3. 管理员新增、修改或删除公告。
4. 系统在首页展示有效公告。
5. 游客、学生、教师浏览公告。

### 4.2 请假业务流程

1. 学生登录系统。
2. 学生填写请假理由、开始时间、结束时间。
3. 系统自动记录申请人和申请时间，状态设为“待审核”。
4. 教师登录后查询本学院学生的待审核申请。
5. 教师审核申请，状态变更为“通过”或“不通过”。
6. 若审核前学生不想继续申请，可撤销申请。
7. 若申请通过，学生返校后进行销假，状态变更为“已销假”。

### 4.3 报修业务流程

1. 学生登录系统。
2. 学生填写报修事由并上传照片。
3. 系统自动记录申请人和申请时间，状态设为“待审核”。
4. 管理员登录后查询待审核报修申请。
5. 管理员审核申请，审核通过时填写维修工手机号。
6. 若审核前学生不想继续申请，可撤销申请。
7. 维修完成后，学生对维修服务进行评分评价。

## 7. 非功能需求

### 7.1 易用性

- 页面结构清晰，按角色展示对应菜单。
- 表单填写应有必填项提示和格式校验。
- 操作成功或失败后应给出明确提示。

### 7.2 安全性

- 用户登录后才能访问受保护功能。
- 密码应加密存储，不能明文保存。
- 后端接口应进行权限校验，不能只依赖前端隐藏按钮。
- 上传照片应校验文件类型和大小。
- 防止普通用户越权查看或修改他人申请。

### 7.3 可靠性

- 关键业务状态变化应保存完整。
- 审核、撤销、销假、评价等操作应记录操作时间。
- 数据库字段应设置必要的非空约束和外键关联。

### 7.4 可维护性

- 建议采用 MVC 分层结构。
- 控制器负责请求处理，服务层负责业务逻辑，数据访问层负责数据库操作。
- 状态值建议使用固定枚举或常量，避免散落硬编码字符串。
